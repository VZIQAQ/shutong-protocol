# 上下文编排器 V2 / Context Orchestrator V2

> **协议版本**：书童协议 V0.2-draft  
> **日期**：2026-07-20  
> **定位**：认知上下文的编排，不是执行上下文的拼接  
> **核心命题**：标签是定位坐标，模型是决策者；没有预算控制，靠智能选择自然控制信息量

---

## 一、一句话定位

**上下文编排器编排的是认知上下文，不是执行上下文。**

它决定"模型现在需要知道什么"，不是"系统现在需要加载什么"。

---

## 二、核心修正：从"机械截断"到"认知选择"

### V0.1 的错误表述（已废弃）

> "截断策略：优先截断中置区，其次近因区，前置区绝不截断"

### V0.2 的正确表述

> **没有预算控制。大脑没有 token 计数器，书童协议也不需要。**
>
> **靠智能选择自然控制信息量。**

```
传统 RAG / MCP：
用户输入 → embedding → 向量相似度计算 → 系统召回 top-k → 机械拼接进上下文
                    ↑
                    └── 系统做计算判断，机械截断尾部

书童协议：
用户输入 → 模型语义判断 → 提取记忆域坐标 → 标签缩小范围 → 模型判断相关性 → 认知性组织上下文
       ↑
       └── 模型做认知判断，注意力聚焦
```

---

## 三、标签系统：定位坐标，不是匹配指令

### 标签的作用

| 误解 | 正确理解 |
|------|---------|
| "标签精确匹配召回记忆" | **标签是书架分类标签，模型是读者，自己决定拿哪本书** |
| "系统根据标签返回所有匹配记忆" | **标签缩小候选范围，召回决策权在模型** |

### 标签体系

```json
{
  "memory_id": "mem_001",
  "content": "用户确认使用JWT进行身份验证",
  "domain": "authentication",
  "topic": "login",
  "type": "confirmed_fact",
  "source": "session_003_round_002",
  "trust_level": "absolute",
  "timestamp": "2026-07-19T10:30:00Z"
}
```

**标签 = {domain, topic, type, source, trust_level}**

**作用：帮模型快速定位到"这个书架区域"。**
**不作用：决定"拿哪本书"。**

### 召回流程

```
用户输入："登录方式改成微信扫码吧"
  → 模型语义判断：
    · 当前记忆域：authentication
    · 相关 topic：login, oauth, wechat, qrcode
    · 需要类型：confirmed_fact（已确认事实）
  → 标签缩小范围：
    · domain=authentication AND topic IN (login, oauth, wechat, qrcode)
    · 候选记忆：15 条
  → 模型判断相关性（逐条）：
    · "JWT 认证" → 相关（历史已确认，需对比）
    · "OAuth2 流程" → 相关（技术方案参考）
    · "微信开放平台配置" → 高度相关
    · "支付宝支付" → 不相关（模型判断丢弃）
  → 召回：3 条
```

**关键：15 条候选 → 模型判断 → 3 条召回。不是系统返回 15 条。**

## 四、四层位阶（不变，但注入逻辑修正）

```
┌─────────────────────────────────────────┐
│  位阶1：系统提示词区（绝不截断）          │
│  · 认知契约（禁止条款+工作流）            │
│  · 身份锚定（模型判断加载哪个专家）        │
│  · 固定注入，每次对话完整加载              │
│  · 体积：~800 tokens（可控）               │
├─────────────────────────────────────────┤
│  位阶2：前置区（热层，绝不截断）          │
│  · 当前会话已确认事实                      │
│  · 已锁定产品包内容                        │
│  · 只追加，不删除，不修改                  │
│  · 体积：~1,500 tokens（30轮以内可控）     │
├─────────────────────────────────────────┤
│  位阶3：中置区（温层/冷层，智能选择）     │
│  · 模型判断召回的记忆片段                   │
│  · 按 trust_level 排序                      │
│  · 模型判断"是否相关"决定取舍               │
│  · 不是系统截断，是模型选择不加载           │
├─────────────────────────────────────────┤
│  位阶4：近因区（智能选择）                │
│  · 用户当前输入                            │
│  · 最近 N 轮对话历史（模型判断 N 的大小）   │
│  · 不是固定 N=10，是模型判断"需要多少上下文" │
└─────────────────────────────────────────┘
```

### 关键修正

| V0.1 表述 | V0.2 修正 |
|-----------|----------|
| "截断策略：优先截断中置区" | **"模型选择：不相关的记忆不加载"** |
| "前置区绝不截断" | **"前置区是已确认事实，必须完整保留"** |
| "中置区按相关性排序，时间截断" | **"中置区按 trust_level 排序，模型判断取舍"** |
| "近因区最近 N 轮" | **"近因区 N 由模型判断"** |

## 五、为什么不会撑爆上下文

### 防线 1：热层整理触发

```
当前会话
  → 每 30 轮：软提醒"建议整理已确认事实"
  → 上下文达 70%：强制终止对话，要求整理
    → 整理完成前，无法继续新对话
```

**这不是截断，是认知节律——像人脑工作记忆饱和后需要睡眠整理。**

### 防线 2：注意力聚焦召回

```
不是"看到 login 就召回所有 authentication 域的记忆"
是"模型判断当前讨论的是认证架构还是登录 UI"
  → 只激活对应的记忆域子集
  → 候选量从 1000 条降到 15 条
```

### 防线 3：模型判断取舍

```
15 条候选记忆
  → 模型逐条判断"是否与当前语境相关"
  → 不相关 → 丢弃（不是截断，是不加载）
  → 相关 → 加载
  → 冲突 → 全部加载，标注"存在冲突"（由梦境整合消化）
```

### 防线 4：项目背景可控

```
三个月项目积累：
  · 愿景 + 角色 + 技术栈 + 领域模型 + 架构决策 + 需求列表 + Spec
  · 总计：几万字 ≈ 10K-20K tokens
  · 当前模型上下文：128K-200K tokens
  · 占比：~10%
```

**在 128K-200K 窗口下，20-30K tokens 的总上下文不会撑爆。**

### 防线 5：梦境整合消化矛盾

```
矛盾记忆：
  · 记忆1：JWT（session_001）
  · 记忆2：改 Session Cookie（session_003）
  · 记忆3：又改回 JWT（session_008）

梦境整合（REFLECT 状态）：
  → 读取原始上下文
  → 补全："session_003 改 Cookie 是因为移动端兼容问题，session_008 改回 JWT 是因为找到了兼容方案"
  → 合并为一条：{content: "JWT（兼容移动端后）", evolution: [...]}
```

**矛盾不是 bug，是需求演化的轨迹。梦境整理后变成版本历史，不再冲突。**

## 六、与 MCP / RAG 的本质区别

| 维度 | MCP / RAG | 书童协议上下文编排 |
|------|----------|------------------|
| **定位** | 执行层上下文拼接 | 认知层上下文组织 |
| **检索机制** | 向量相似度 / 全量暴露 | **模型判断 + 标签定位** |
| **注入方式** | 机械拼接（top-k 截断） | **认知性组织（模型选择取舍）** |
| **上下文成本** | O(n) 工具/记忆数量 | **O(1) 与记忆总量无关** |
| **信息保真** | 摘要 / 向量压缩 | **原文完整注入（零压缩）** |
| **谁控制信息量** | 系统（截断阈值） | **模型（注意力聚焦）** |

### MCP 的上下文爆炸数据

| 工具数量 | 消耗 Tokens | 窗口占比 |
|---------|------------|---------|
| 26 个 | ~4,600 | ~2% |
| 40 个 | ~55,000 | ~27% |
| 100+ 个 | ~143,000 | ~72% |

**书童协议通过"注意力聚焦 + 模型选择取舍"，从根本上避免这种爆炸。**
