# 模型嵌入判断 / Model-Embedded Judgment

> **协议版本**：书童协议 V0.2-draft  
> **日期**：2026-07-20  
> **定位**：书童协议的第一性原理  
> **核心命题**：模型的语义判断能力嵌入代码的决策节点，替代关键词匹配和硬编码规则

---

## 一、问题：传统代码的判断瓶颈

### 关键词列表的宿命

```python
# 传统方案：永远膨胀的关键词列表
STOP_KEYWORDS = ["就这样", "先这样", "够了", "别问了", ...]
if any(kw in user_input for kw in STOP_KEYWORDS):
    stop()
```

| 输入 | 关键词匹配 | 结果 |
|------|-----------|------|
| "就这样" | ✅ 触发 | stop |
| "我不想就这样结束" | ❌ 误触发 | stop（否定词被忽略） |
| "赶紧给我出方案别墨迹了" | ❌ 漏掉 | continue（语义被忽略） |

**关键词列表的维护成本是 O(n)，且永远有边界情况。**

### if-else 规则树的崩溃

```python
# 传统方案：永远分叉的规则树
if "停止" in input:
    if "不想" in input:
        if "结束" in input:
            ...
```

规则越多，冲突越多，维护越难。**规则编写者的智能上限 = 系统的智能上限。**

## 二、模型嵌入判断：让模型做语义理解，代码做确定性校验

### 核心设计

```
用户输入
    ↓
【模型嵌入判断节点】（概率性）
  · 本地小模型（7B）理解语义
  · 返回判断结果（intent / state / memory_domain）
    ↓
【规则层校验】（确定性）
  · 结构是否完整？
  · 来源标注是否合规？
  · 状态切换是否合法？
    ↓
【执行或追问】
  · 校验通过 → 执行
  · 校验失败 → 重新输出 / 追问用户
```

### 关键区分

| 层级 | 谁做 | 做什么 | 性质 |
|------|------|--------|------|
| **判断层** | 模型 | 语义理解、意图识别、状态判断、记忆相关性评估 | 概率性 |
| **校验层** | 代码 | 结构检查、来源标注扫描、状态合法性、执行拦截 | 确定性 |
| **执行层** | 系统（经确认后） | 调用工具、写入文件、输出结果 | 绝对确定性 |

**模型做判断，不做决策。决策需要校验 + 用户确认。**

---

## 三、为什么不是"模型自治"

### "自治"在 AI 语境下的歧义

| 概念 | Agent 循环的"自治" | 书童协议的"模型嵌入判断" |
|------|-------------------|------------------------|
| **范围** | 行动自由（拆解任务、调用工具、自主执行） | 认知判断自由（理解语义、判断状态、评估相关性） |
| **执行权** | 有直接执行权 | **无执行权**，需校验 + 用户确认 |
| **用户角色** | 旁观者 | **校准者** |
| **本质** | 让模型更像人（自主行动） | 让模型更诚实（暴露认知状态） |

**书童协议不是"模型自治"，是"模型嵌入判断"——判断归模型，执行归用户。**

## 四、判断节点的具体应用

### 4.1 意图识别

```python
# 模型嵌入判断
intent = await _recognize_intent(user_input, conversation_history)
# → "stop" / "continue" / "direct_write" / "show" / "skip"

# 不是关键词匹配，是语义理解
# "我不想就这样结束" → continue（否定词反转）
# "赶紧给我出方案别墨迹了" → stop（语义推断）
```

### 4.2 状态切换判断

```python
# 模型判断当前该进入什么状态
state = await _judge_state(current_context, user_input)
# → THINK / QUERY / ANSWER / EXECUTE / REFLECT

# 不是系统硬编码，是模型语义判断
# "分析一下这个架构" → THINK
# "登录方式你倾向哪种？" → QUERY（信息不足）
# "确认" → ANSWER（基于已确认事实）
```

### 4.3 记忆域激活判断

```python
# 模型判断当前语境激活哪个记忆域
domain = await _judge_memory_domain(user_input, current_topic)
# → {domain: "authentication", related: ["login", "oauth", "jwt"]}

# 不是系统计算相似度，是模型语义判断
# "登录方式" → 激活 authentication 域
# "支付回调" → 激活 payment 域
```

### 4.4 记忆相关性判断

```python
# 模型判断召回的记忆是否与当前语境相关
for memory in candidate_memories:
    relevance = await _judge_relevance(memory, current_context)
    # → relevant / irrelevant

# 不是系统计算 embedding 相似度，是模型语义判断
# "JWT 过期策略"在讨论登录时 → relevant
# "JWT 过期策略"在讨论支付时 → irrelevant（模型判断）
```

## 五、边界：模型判断什么，不判断什么

### 模型判断的（语义层面）

- ✅ 用户意图是什么（语义理解）
- ✅ 当前该进入什么状态（认知判断）
- ✅ 哪个记忆域与当前语境相关（语义关联）
- ✅ 召回的记忆是否相关（语义评估）
- ✅ 追问措辞如何生成（创造性采样）

### 模型不判断的（确定性层面）

- ❌ 结构是否完整（规则引擎扫描）
- ❌ 来源标注是否合规（正则匹配）
- ❌ 是否允许执行（规则层拦截）
- ❌ 数学计算是否正确（代码计算）
- ❌ 权限是否合法（代码校验）

## 六、与现有方案的关系

| 现有方案 | 定位 | 与书童协议的关系 |
|---------|------|----------------|
| **Semantic Router** | 执行层路由 | 书童协议在其之前做语义判断，决定"是否需要路由" |
| **MCP 工具发现** | 工具通信协议 | 书童协议判断"当前语境需要哪些工具"，MCP 负责实际调用 |
| **关键词匹配** | 传统方案 | 书童协议替代的对象 |
| **向量检索（RAG）** | 记忆召回 | 书童协议用模型判断替代向量相似度计算 |

## 七、核心主张

> **模型的语义判断能力应该嵌入代码的每一个决策节点，替代关键词匹配和硬编码规则。**
>
> **但判断之后不是直接执行，是进入规则校验和用户确认。**
>
> **模型有判断权，没有执行权。**
