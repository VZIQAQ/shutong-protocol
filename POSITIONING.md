# 书童协议定位声明 / ShuTong Protocol Positioning

> **协议版本**：书童协议 V0.1  
> **日期**：2026-07-20  
> **核心命题**：对齐后的理解都是记忆。记忆是执行的唯一依据。

---

## 一、一句话定位

**书童协议是认知层的人机对齐协议。**

它不定义执行层。它只定义一件事：**模型和用户对需求的理解如何达成一致，并以记忆的形式固化下来。**

执行层拿到这些记忆，按自己的方式执行即可。

---

## 二、核心命题：对齐后的理解都是记忆

### 传统误区

人们习惯把"理解"和"执行"分开：
- 理解 = 脑子里想明白了
- 执行 = 动手做出来

但在人机协作中，**"理解"如果只在模型权重里，用户看不见、摸不着、无法校验。**

### 书童协议的立场

> **对齐后的理解必须固化为记忆。**
> 
> 记忆不是执行的附属品，记忆是执行的前提。

```
用户输入
    ↓
模型语义判断（选择层）
    ↓
追问-校准-确认（认知对齐过程）
    ↓
【对齐后的理解】→ 固化为记忆（温层）
    ↓
执行层读取记忆 → 按自己的方式执行
    ↓
执行结果
```

**产品包（.shutong/ 目录）只是记忆的一种表现形式。**

它可以是：
- Markdown 文件（产品包）
- JSON Schema（结构化需求）
- 数据库记录（确认事实表）
- 甚至是一段自然语言摘要

**形式不重要，重要的是：它是经过用户确认的记忆。**

## 三、认知层与执行层的边界

### 一刀切开

```
┌─────────────────────────────────────────┐
│           认知层（书童协议）              │
│                                         │
│  核心问题："我理解对了吗？"              │
│                                         │
│  · 意图识别（语义判断）                   │
│  · 假设暴露（认知显化）                   │
│  · 追问确认（QUERY 状态）                │
│  · 记忆召回（模型判断 + 标签定位）        │
│  · 理解固化（确认事实写入温层）           │
│                                         │
│  产出：记忆（已对齐、已确认、可追溯）     │
└─────────────────────────────────────────┘
                    ↓
         【记忆是执行的唯一输入】
                    ↓
┌─────────────────────────────────────────┐
│           执行层（Agent 框架 / 工具）      │
│                                         │
│  核心问题："我怎么完成任务？"            │
│                                         │
│  · 任务拆解                              │
│  · 工具调用                              │
│  · 代码生成                              │
│  · 自主重试                              │
│  · 结果观察                              │
│                                         │
│  输入：记忆（来自书童协议）                │
│  产出：执行结果                           │
└─────────────────────────────────────────┘
```

### 书童协议不管的事

- ❌ 任务拆解（执行层自己做）
- ❌ 工具调用（执行层自己做）
- ❌ 自主重试（执行层自己做）
- ❌ 代码生成（执行层自己做）
- ❌ 结果优化（执行层自己做）

### 书童协议管的事

- ✅ 在任务拆解前，确保"要拆什么"已经被对齐
- ✅ 在工具调用前，确保"用户想要什么结果"已经被确认
- ✅ 在代码生成前，确保"用户确认的需求是什么"已经被记录
- ✅ 在执行出错时，提供"对齐时的原始记忆"用于追溯

**执行层不需要理解用户，它只需要读取记忆。**

## 四、记忆的三层结构

### 热层（当前会话）

- 原始对话（未整理）
- 每 30 轮或上下文达 70% 触发整理
- 整理后写入温层

### 温层（已对齐的理解）

- 用户确认的事实（原文，零压缩）
- 追问-确认链（时间戳、来源、原话对照）
- 产品包（.shutong/ 目录下的文件）
- **这是执行层读取的唯一记忆来源**

### 冷层（原始存档）

- 完整对话历史（按 session ID 索引）
- 仅在需要追溯原始上下文时读取
- 不直接用于执行

### 执行层只读温层

```
执行层启动
    ↓
读取温层记忆（已对齐的理解）
    ↓
按记忆执行任务
    ↓
执行过程中遇到问题？
    ↓
回查温层："用户当时确认的是什么？"
    ↓
继续执行或报错
```

**执行层不需要追问用户，不需要做认知对齐——这些已经在书童协议里完成了。**

## 五、与现有框架的关系

### 书童协议是执行层的"前置对齐层"

```
用户
    ↓
书童协议（认知层）
    → 追问、确认、锁定
    → 产出：记忆（温层）
    ↓
执行层（Agent 框架 / 编程 AI / 工具链）
    → 读取记忆
    → 拆解、执行、输出
    ↓
结果
```

### 具体对应

| 现有框架 | 层级 | 与书童协议的关系 |
|---------|------|----------------|
| **ReAct / AutoGPT / Devin** | 执行层 | 书童协议为其提供"已对齐的记忆"作为输入 |
| **MCP（Model Context Protocol）** | 工具通信层 | 书童协议不替代 MCP，在其之上做认知对齐 |
| **A2A（Agent-to-Agent）** | Agent 通信层 | 书童协议可为多 Agent 提供统一的对齐记忆 |
| **CHAP** | 协作层 | 与书童协议互补，CHAP 管流程，书童管理解 |
| **Semantic Router** | 执行层路由 | 书童协议在其之前做意图语义判断 |

### 关键区分

> **书童协议不是"另一个 Agent 框架"，是"所有 Agent 框架的前置对齐层"。**
>
> 就像 TCP/IP 是 HTTP 的底层协议，书童协议是 Agent 框架的底层对齐协议。

## 六、为什么必须分层

### 混在一起的代价

如果把认知和执行混在一起：

```
用户：做个电商后台
    ↓
Agent：拆解 → 写代码 → 调 API → 出结果
    ↓
用户：这不是我要的
    ↓
Agent：重试 → 还是不对
    ↓
用户：你理解错了
    ↓
Agent：但我已经执行了...

问题：错误在执行后才被发现，纠正成本 = 全部返工
```

### 分层后的收益

```
用户：做个电商后台
    ↓
书童协议：追问 → 确认 → 锁定记忆
    ↓
用户：确认，这就是我要的
    ↓
执行层：基于锁定的记忆拆解 → 写代码 → 调 API
    ↓
出错？→ 回查记忆："用户确认的是 A，你做了 B"
    ↓
责任清晰：执行层违反了记忆，还是记忆本身有歧义？

收益：错误在认知层就被拦截，执行层只负责"按记忆执行"
```

**对齐成本 << 返工成本。**
