## 一、架构层：系统组件

### 1.1 系统架构图

```
┌─────────────────────────────────────────────────────────────────┐
│                         用户交互层                                │
│  · 自然语言对话输入（纯对话，不弹卡片）                            │
│  · 文件草案确认（唯一结构化交互）                                  │
│  · 产品包锁定确认                                                 │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                         对话引擎                                  │
│  · 接收用户输入                                                  │
│  · 调用上下文编排器组装上下文                                     │
│  · 调用LLM生成回复（认知显化，自然语言）                          │
│  · 提交规则层检验（结构检测，非语义判断）                          │
│  · 根据检验结果：追问 / 预览 / 执行拦截                           │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                         上下文编排器                             │
│  · 四层位阶组装（系统/前置/中置/近因）                            │
│  · 记忆域检索（热/温/冷）                                        │
│  · 片段编码（结构化+元数据）                                       │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                         规则引擎                                  │
│  · 结构检验：LLM输出是否含假设清单/盲区清单                       │
│  · 来源标注检验：正则匹配"来源：纯猜测"                           │
│  · 轮次提醒：≥N轮时追问末尾追加"可随时喊停"提示                   │
│  · 执行拦截：未确认→禁止输出产品包                                │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                         记忆域                                    │
│  · 热层：当前会话前置区（实时读写）                                │
│  · 温层：项目级记忆（会话结束沉淀）                                │
│  · 冷层：技能库/行业知识（手动维护）                               │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                         产品包管理器                               │
│  · 产品包组装（.shutong/ 目录）                                   │
│  · 锁定机制（文件状态机：empty→drafting→confirmed→locked）        │
│  · 变更协议（用户显式发起→重新对齐→重新锁定）                      │
│  · 编程AI接口输出                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 组件职责边界

| 组件 | 职责 | 禁止做的事 |
|------|------|-----------|
| **对话引擎** | orchestrate 整个流程，调用各组件 | 不自己做认知判断，只调用规则引擎 |
| **上下文编排器** | 按规则组装上下文，检索记忆 | 不判断"哪些记忆重要"，只按位阶规则组装 |
| **规则引擎** | 结构检测：LLM输出是否符合协议格式 | 不做语义判断，不判断"内容是否正确" |
| **记忆域** | 结构化存储，原子读写 | 不压缩，不推断，只存原文+元数据 |
| **产品包管理器** | 格式化输出，锁定管理，变更审计 | 不修改已锁定内容，变更必须经用户显式发起 |

### 2.2 片段编码规范

所有进入上下文的记忆片段必须编码为结构化文本：

```json
{
  "fragment_id": "frag_001",
  "content": "用户确认使用JWT进行身份验证",
  "source": "session_003_round_002",
  "source_type": "user_confirmed",
  "timestamp": "2026-07-19T10:30:00Z",
  "topic": "authentication",
  "trust_level": "absolute",
  "expiry": null
}
```

**编排规则（确定性）：**
1. 前置区片段：source_type="user_confirmed" 的按时间序全量注入
2. 中置区片段：按topic相关性召回，按trust_level排序，时间截断
3. 截断策略：优先截断中置区，其次近因区，前置区绝不截断
4. 冲突处理：同一topic存在矛盾片段时，全部注入并标注"存在冲突"，由模型在认知显化中暴露

## 四、追问引擎：状态机与触发规则

### 4.1 追问状态机

```
                    ┌─────────────┐
                    │   START     │
                    └──────┬──────┘
                           │ 用户输入
                           ▼
                    ┌─────────────┐
                    │ COGNITION   │ 模型生成认知显化（对话形式）
                    │  (认知显化)  │
                    └──────┬──────┘
                           │
                           ▼
                    ┌─────────────┐
                    │  RULE_CHECK │ 规则引擎检验（结构检测）
                    │  (规则检验)  │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
     存在纯猜测/阻断盲区  轮次≥N提醒   无问题
              │            │            │
              ▼            ▼            ▼
        ┌─────────┐ ┌─────────┐ ┌─────────┐
        │CLARIFY  │ │CLARIFY  │ │PREVIEW  │
        │(正常追问)│ │(带提醒) │ │(预览)   │
        │         │ │追问末尾  │ │         │
        │         │ │追加提示  │ │         │
        └────┬────┘ └────┬────┘ └────┬────┘
             │           │           │
             │    用户确认/修改    用户锁定
             │           │           │
             └───────────┴───────────┘
                           │
                           ▼
                    ┌─────────────┐
                    │   LOCKED    │ 产品包锁定
                    │  (已锁定)    │
                    └──────┬──────┘
                           │
                           ▼
                    ┌─────────────┐
                    │  EXECUTION  │ 交给编程AI
                    │  (执行层)    │
                    └─────────────┘
```

**轮次≥N时的行为说明：**

这不是独立的状态转换，而是**CLARIFY状态下的分支行为**。当同一话题的追问轮次达到N时，书童在追问内容的末尾追加一句提醒：

> "关于[话题]还有X个未确认项，你可以继续补充，也可以随时说'就这样'让我先出预览。"

**为什么是软提醒而非硬限制：**
- 目标用户是"想要做产品的用户"，需要充分对齐
- 复杂需求确实需要多轮才能说清楚
- 硬限制会打断深度思考，软提示让用户自己决定"够了吗"

### 4.2 追问触发规则（已验证）

> **V0.1 实现**：意图识别已重构为本地模型语义判断（`_recognize_intent()`），零关键词匹配。
> 规则引擎（`rule_engine.py`）只做文本模式扫描，不调用 LLM。

**实际实现流程：**

```
用户输入
  → _recognize_intent()（本地 Ollama 模型，读完整对话历史，返回 5 种意图之一）
  → stop → 有 pending_draft 则确认写入，无则 _force_preview() 强制出预览
  → direct_write → fb.write_file() 直接写磁盘
  → show → 展示草案卡片
  → skip → 跳过草案
  → continue → _llm_cognition_flow()（LLM 认知显化 + rule_engine 结构校验）
```

**规则引擎校验逻辑（`rule_engine.py`）：**
- 扫描 LLM 输出，检查是否包含"假设清单""盲区清单"结构
- 正则匹配"来源：纯猜测" → 强制追问
- 正则匹配"是否阻断：是" → 强制追问
- 不做语义判断，只做文本模式匹配

### 4.3 轮次计数规则（已验证）

**V0.1 实现**：`session.py` 中 `clarify_counter: dict[str, int]` 按 topic 维度计数。

- 轮次 < 3：正常追问，不追加提示
- 轮次 >= 3：追问末尾追加提醒："你也可以随时说「就这样」让我先出预览"
- 用户说"就这样"（intent=stop）：`_force_preview()` 强制出预览，标注"用户主动停止追问"
- 用户说"确认"（intent=direct_write）：`fb.write_file()` 写入磁盘

## 五、产品包：结构与锁定机制

### 5.1 产品包结构（.shutong/ 目录）

MVP阶段产品包 = `.shutong/` 目录下的Markdown文件，不需要JSON Schema：

```
.shutong/
├── README.md              # 目录索引
├── 01-vision.md           # 项目愿景
├── 02-persona.md          # 用户角色
├── 03-techstack.md        # 技术栈
├── 04-codestyle.md        # 代码风格
├── 05-domain.md           # 领域模型
├── 06-architecture.md     # 架构决策
├── 07-patterns.md         # 模式约定
├── 08-lessons.md          # 经验教训
├── backlog.md             # 需求池
├── specs/                 # Spec文档
│   └── round-001-xxx.md
└── handoffs/              # 交接文档
    └── round-001-handoff.md
```

每个文件有独立状态：
- `empty`：未完善
- `drafting`：追问中
- `pending_confirm`：草案待用户确认
- `confirmed`：已确认写入
- `locked`：Spec已锁定

### 5.2 锁定机制

锁定通过文件frontmatter状态机实现：

```markdown
---
status: LOCKED
round: 1
feature: coupon
created_at: 2026-07-18T14:00:00+08:00
locked_at: 2026-07-18T14:30:00+08:00
---
```

锁定规则：
- 只有 `confirmed` 状态的文件内容才能进入锁定的Spec
- 锁定后不可变，修改需先解锁
- 产品包的完整性由文件系统保证，不需要签名防篡改（MVP阶段）

### 5.3 变更协议（REVISED）

**锁定不是永久封印，是"变更需经重新对齐"的协议。**

```
变更流程：
1. 用户说"我要改XX" → 书童将对应文件标记为 REVISED
2. 重新进入认知显化 → 追问 → 确认流程
3. 变更后的文件重新标记为 confirmed → locked
4. 原锁定版本保留历史（不删除，用于审计追溯）
5. 审计链记录：何时、因何、由谁变更

禁止：
- 禁止书童或编程AI擅自修改locked文件
- 禁止绕过重新对齐流程直接修改
- 禁止删除历史版本（保留审计链）
```

### 5.4 编程AI接收协议

```
【编程AI接收规范】

输入：书童产品包（.shutong/ 目录，所有文件status必须为confirmed或locked）

处理规则：
1. 读取 01-vision.md：理解项目目标
2. 读取 05-domain.md：理解业务实体和规则
3. 读取 03-techstack.md：遵守技术选型约束
4. 读取 specs/round-xxx.md：严格按Spec实现
5. 读取 04-codestyle.md：遵守编码规范
6. 禁止行为：
   - 禁止绕过产品包直接向用户提问
   - 禁止修改用户已确认的设计决策
   - 禁止引入Spec中未提及的依赖

输出：
- 执行产物（代码/配置/文档）
- 偏差报告（如有与产品包不一致的技术决策）
- 技术冲突反馈（如有无法实现的需求）
```
